# Webhooks

Los webhooks permiten recibir notificaciones HTTP en tiempo real cuando ocurren eventos en tus integraciones de JAAK. En lugar de consultar periódicamente la API, tu servidor recibe automáticamente los datos cuando el evento ocurre.

## Cómo funcionan

```
1. Configuras una URL de webhook en tu dashboard de la compañía
2. Ocurre un evento (ej: sesión KYC completada)
3. JAAK envía un POST a tu URL con los datos del evento
4. Tu servidor procesa el evento y responde con 200 OK
```

<Callout type="info" title="Requisitos de tu endpoint">
Tu endpoint debe ser accesible públicamente vía HTTPS y responder en menos de 10 segundos. URLs con HTTP o localhost no son aceptadas en producción.
</Callout>

---

## Configuración

Los webhooks se configuran a nivel de **compañía** (no por sesión individual). Cada compañía puede tener un webhook por cada flujo de producto (KYC, Signa, etc.).

### En el Dashboard

1. Accede a **Configuración** → **Compañía** → **Webhooks** en tu dashboard
2. En la sección **KYC Tradicional → Sesión finalizada**:
   - Ingresa la **URL** de tu endpoint (HTTPS)
   - Opcionalmente, configura una **Auth Key** (ver más abajo)
3. Guarda los cambios. La configuración aplica inmediatamente a todas las sesiones nuevas y a las que finalicen después del cambio.

<Callout type="warning" title="Una sola URL por compañía">
Hoy la compañía tiene una única URL de webhook para el evento de KYC finalizado. Si necesitas enrutar a varios destinos (ej. staging vs prod), usa un proxy o tu propio router.
</Callout>

## Webhook de sesión KYC finalizada

JAAK dispara este webhook **una sola vez por sesión** cuando se ejecuta el endpoint `POST /api/v1/kyc/session/finish`. Es el único webhook que se emite para el flujo KYC.

### Headers que envía JAAK

```http
POST https://tu-servidor.com/webhook/kyc
Content-Type: application/json
Api-AuthKey: <tu-auth-key-en-texto-plano>
```

| Header | Descripción |
|---|---|
| `Content-Type` | Siempre `application/json` |
| `Api-AuthKey` | El valor exacto que configuraste en el dashboard, en texto plano. Solo se envía si configuraste Auth Key. |

<Callout type="info" title="Si no configuraste Auth Key">
Si dejaste el campo Auth Key vacío en el dashboard, el header `Api-AuthKey` **no se envía**. Tu endpoint debe entonces aceptar la petición sin credenciales o validar por otro mecanismo (IP allowlist, mutual TLS, etc.).
</Callout>

### Estructura del payload

```json
{
  "id": "69ed18794fc4af88ecacf4b0",
  "shortKey": "AvBuahf",
  "startedAt": "2026-04-25T10:30:00Z",
  "endedAt": "2026-04-25T10:35:00Z",
  "duration": "5.00m",
  "validation": "Automatic",
  "stage": "approved",
  "scores": {
    "liveness": 96.1,
    "document": 98.2,
    "oneToOne": 94.7,
    "blacklist": 100.0,
    "total": 95.5,
    "status": "PASS",
    "livenessPass": true,
    "documentPass": true,
    "oneToOnePass": true
  },
  "detailsURL": "https://services.api.jaak.ai/api/v1/kyc/session/69ed18794fc4af88ecacf4b0"
}
```

| Campo | Tipo | Descripción |
|---|---|---|
| `id` | string | MongoDB ObjectID de la sesión (24 caracteres hexadecimales). |
| `shortKey` | string | Identificador corto de 7 caracteres usado durante el flujo. |
| `startedAt` | ISO 8601 | Timestamp en que el usuario inició el flujo. |
| `endedAt` | ISO 8601 | Timestamp en que se llamó a `/finish`. |
| `duration` | string | Tiempo total de la sesión, formato `M.MMm` (minutos con dos decimales). |
| `validation` | enum | `Automatic`, `Manual`, `NeedReview`. Indica cómo se resolvió la sesión. |
| `stage` | enum | `approved`, `rejected`, `expired`, `in_review`, etc. Resultado final. |
| `scores` | object | Puntuaciones por componente. Ver detalle abajo. |
| `detailsURL` | string | URL absoluta al endpoint de JAAK con los detalles completos de la sesión. Usa tu API Key para consultarlo. |

### Estructura de `scores`

| Campo | Tipo | Descripción |
|---|---|---|
| `liveness` | number | Score 0-100 de la prueba de vida. |
| `document` | number | Score 0-100 de la verificación del documento. |
| `oneToOne` | number | Score 0-100 del face match (selfie vs foto del documento). |
| `blacklist` | number | Score 0-100 de las consultas a listas. |
| `total` | number | Score agregado 0-100. |
| `status` | enum | `PASS`, `FAIL`, `NEED_REVIEW`. |
| `livenessPass` | boolean | Si el liveness pasó el umbral configurado. |
| `documentPass` | boolean | Si el documento pasó el umbral. |
| `oneToOnePass` | boolean | Si el face match pasó el umbral. |

### Datos personales

<Callout type="info" title="El webhook NO incluye datos personales">
El payload solo trae identificadores y scores. Para obtener nombre, fecha de nacimiento, CURP, dirección, imágenes, etc., consulta el endpoint indicado en `detailsURL` con tu API Key:

```bash
curl -H "Authorization: Bearer YOUR_API_KEY" \
  "https://services.api.jaak.ai/api/v1/kyc/session/69ed18794fc4af88ecacf4b0"
```
</Callout>

## Verificación con Auth Key

A diferencia de muchos servicios, JAAK **no firma los webhooks con HMAC**. La autenticación es por header de string compartido (`Api-AuthKey`).

### Implementación

#### Node.js (Express)

```javascript
const express = require('express');
const app = express();
app.use(express.json());

// La misma Auth Key que configuraste en el dashboard
const EXPECTED_AUTH_KEY = process.env.JAAK_WEBHOOK_AUTH_KEY;

app.post('/webhook/kyc', (req, res) => {
  // Validar Auth Key (solo si la configuraste)
  if (EXPECTED_AUTH_KEY) {
    const receivedKey = req.headers['api-authkey'];
    if (receivedKey !== EXPECTED_AUTH_KEY) {
      return res.status(401).send('Invalid auth key');
    }
  }

  // Procesar el evento
  const event = req.body;
  console.log('Sesión KYC finalizada:', event.id, '-', event.stage);

  // Responder rápido
  res.status(200).send('OK');

  // Hacer trabajo pesado de forma asíncrona
  processSessionAsync(event);
});

async function processSessionAsync(event) {
  if (event.stage === 'approved') {
    // Consultar detalles completos
    const details = await fetch(event.detailsURL, {
      headers: { Authorization: `Bearer ${process.env.JAAK_API_KEY}` }
    }).then(r => r.json());

    // ... tu lógica de negocio
  }
}
```

#### Python (Flask)

```python
import os
from flask import Flask, request, jsonify
import requests

app = Flask(__name__)
EXPECTED_AUTH_KEY = os.environ.get('JAAK_WEBHOOK_AUTH_KEY')
JAAK_API_KEY = os.environ['JAAK_API_KEY']

@app.route('/webhook/kyc', methods=['POST'])
def webhook_kyc():
    # Validar Auth Key
    if EXPECTED_AUTH_KEY:
        received = request.headers.get('Api-AuthKey', '')
        if received != EXPECTED_AUTH_KEY:
            return jsonify({"error": "Invalid auth key"}), 401

    event = request.get_json()

    # Responder rápido
    response = jsonify({"received": True})
    response.status_code = 200

    # Procesar en background
    if event['stage'] == 'approved':
        details = requests.get(
            event['detailsURL'],
            headers={'Authorization': f'Bearer {JAAK_API_KEY}'}
        ).json()
        # ... tu lógica de negocio

    return response
```

#### Go

```go
package main

import (
    "encoding/json"
    "net/http"
    "os"
)

var expectedAuthKey = os.Getenv("JAAK_WEBHOOK_AUTH_KEY")

type KycWebhookEvent struct {
    ID         string  `json:"id"`
    ShortKey   string  `json:"shortKey"`
    StartedAt  string  `json:"startedAt"`
    EndedAt    string  `json:"endedAt"`
    Duration   string  `json:"duration"`
    Validation string  `json:"validation"`
    Stage      string  `json:"stage"`
    Scores     Scores  `json:"scores"`
    DetailsURL string  `json:"detailsURL"`
}

type Scores struct {
    Liveness     float64 `json:"liveness"`
    Document     float64 `json:"document"`
    OneToOne     float64 `json:"oneToOne"`
    Blacklist    float64 `json:"blacklist"`
    Total        float64 `json:"total"`
    Status       string  `json:"status"`
    LivenessPass bool    `json:"livenessPass"`
    DocumentPass bool    `json:"documentPass"`
    OneToOnePass bool    `json:"oneToOnePass"`
}

func webhookHandler(w http.ResponseWriter, r *http.Request) {
    if expectedAuthKey != "" {
        if r.Header.Get("Api-AuthKey") != expectedAuthKey {
            http.Error(w, "Invalid auth key", http.StatusUnauthorized)
            return
        }
    }

    var event KycWebhookEvent
    if err := json.NewDecoder(r.Body).Decode(&event); err != nil {
        http.Error(w, "Bad payload", http.StatusBadRequest)
        return
    }

    // Procesar (de forma idempotente — usa event.ID como key)
    go processSessionAsync(event)

    w.WriteHeader(http.StatusOK)
    w.Write([]byte("OK"))
}
```

<Callout type="warning" title="Recomendaciones de seguridad">
- Usa una Auth Key de **al menos 32 caracteres** con letras, números y símbolos.
- Guárdala en variables de entorno o un secret manager — nunca en código.
- Si tu endpoint está expuesto públicamente, complementa la validación con allowlist de IPs.
- HTTPS es obligatorio. JAAK no envía webhooks a HTTP plano.
</Callout>

## Política de reintentos

<Callout type="warning" title="No hay reintentos automáticos">
JAAK realiza **un solo intento** de envío del webhook con timeout de **10 segundos**. Si tu endpoint responde con error, demora más de 10s, o no responde, el webhook se pierde.

Estamos trabajando en una política de reintentos con backoff exponencial. Por ahora, recomendamos:

- Mantener tu endpoint disponible y rápido (< 1s idealmente).
- Implementar el patrón "responder rápido, procesar después" (ver código de ejemplo arriba).
- Usar el endpoint `GET /api/v1/kyc/session/:id` con polling como fallback para sesiones que no recibiste por webhook.
</Callout>

---

## Mejores prácticas

### 1. Responde rápidamente

Tu endpoint debe responder con `200 OK` en menos de 10 segundos. Si necesitas hacer procesamiento pesado, hazlo de forma asíncrona después de responder.

### 2. Implementa idempotencia

Aunque hoy no hay reintentos automáticos, si en el futuro los hay tu endpoint debería estar preparado. Usa el campo `id` (sessionID) para evitar procesar duplicados:

```javascript
async function processEvent(event) {
  const exists = await db.processedSessions.findOne({ sessionId: event.id });
  if (exists) return;

  await handleEvent(event);
  await db.processedSessions.insertOne({ sessionId: event.id, processedAt: new Date() });
}
```

### 3. Consulta detalles solo cuando sea necesario

El payload trae lo esencial (decisión, scores). Solo consulta `detailsURL` cuando vayas a usar los datos personales (alta de cliente, persistir en tu DB, etc.).

### 4. Registra todos los webhooks recibidos

Guarda un log para debugging:

```javascript
app.post('/webhook/kyc', async (req, res) => {
  await db.webhookLogs.insertOne({
    receivedAt: new Date(),
    headers: req.headers,
    body: req.body
  });

  res.status(200).send('OK');
  // ... procesar
});
```

## Solución de problemas

### Mi endpoint no recibe webhooks

1. Confirma que la URL en el dashboard sea correcta y use `https://`.
2. Verifica que tu firewall permita conexiones entrantes desde IPs de JAAK.
3. Asegúrate de que el endpoint responda en menos de 10 segundos.
4. Si configuraste Auth Key, confirma que tu validación coincida exactamente con el valor del dashboard (case-sensitive, sin espacios extra).

### El webhook no llega aunque la sesión se finalizó

1. Revisa los logs de tu endpoint — puede haber respondido un 4xx/5xx accidentalmente.
2. Confirma que el endpoint `/api/v1/kyc/session/finish` se haya ejecutado exitosamente (no por timeout o error previo).
3. Como fallback, consulta `GET /api/v1/kyc/session/:id` con polling.

### Recibí el webhook pero no tengo los datos del usuario

El payload no incluye datos personales por diseño. Consulta los detalles completos con tu API Key:

```bash
curl -H "Authorization: Bearer YOUR_API_KEY" "$detailsURL"
```
